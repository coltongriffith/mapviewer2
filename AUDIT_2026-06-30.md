# Mapviewer2 — Codebase & UI/UX Audit

Date: 2026-06-30

## 1. Summary

Mapviewer2 is a React 18 + Vite 5 single-page app for building mining-claims maps, using Leaflet 1.9 for the map canvas, Turf/proj4/d3-geo for geospatial math, jsPDF/jszip for export, and Supabase (Postgres + auth) for storage and accounts. It's pure JavaScript (no TypeScript), has no ESLint config, and has no automated tests.

Overall the application logic is solid — error handling around file imports, env-var guards around Supabase, and the existing `ErrorBoundary` are all done correctly. The main risks are: (a) a handful of small, genuine bugs around HTML escaping and silent error swallowing, (b) accessibility gaps on a few interactive controls, and (c) the complete absence of linting/tests/type-checking, which is the biggest long-term risk to code quality as the app grows (`App.jsx` alone is ~5,200 lines).

This report also reconciles against the prior audit, `BUG_UIUX_AUDIT.md` (dated 2026-03-31). 144 commits have landed since that doc was written, and **6 of its 10 items are already fixed** — see §2.

Severity key: 🔴 High · 🟠 Medium · 🟡 Low

## 3. New functional / logic findings

### 🟠 3.1 Unescaped owner name in dissolved-claims popup/tooltip HTML
**File:** `src/App.jsx` (dissolved-by-owner rendering branch, around lines 1826–1850)

When claims are grouped/dissolved by owner, the code builds Leaflet tooltip and popup markup directly from `displayName` (derived from registry owner names) without escaping:

```js
lyr.bindTooltip(`<strong>${displayName}</strong><br/>...`, ...)
...
el.innerHTML = `<div class="acp-owner">${displayName}</div>` + ...
```

This is inconsistent with the sibling, non-dissolved code path a few lines below, which uses `claimPopupRowsHtml`/`claimTooltipHtml` from `utils/claimInfo.js` — both of which already escape via a local `esc()` helper. An owner name containing `<`, `&`, or `"` (plausible in real-world government registry data — legal entity names do contain ampersands, e.g. "Smith & Sons Mining") will render incorrectly today, and in the worst case this is an HTML-injection-shaped bug if the registry data were ever attacker-influenced.

**Fix applied in this pass:** exported the existing `esc()` helper from `claimInfo.js` and applied it everywhere `displayName` is interpolated into `innerHTML`/`bindTooltip` strings in the dissolved branch. (`text: displayName` passed into `addCalloutAtAnchor` was left alone — that path renders as a normal React child in `CalloutsOverlay.jsx`, not raw HTML, so it was already safe.)

### 🟡 3.2 Silent failures in telemetry/search promise chains
**Files:** `src/App.jsx` (`logView()`, ~lines 925–938), `src/hooks/useClaims.js` (`searchOtherProvinces`, ~lines 48–65)

- `logView()` fires Supabase `page_views` inserts and silently drops the error if both the primary and fallback insert fail.
- `searchOtherProvinces` uses `Promise.allSettled` (good — avoids one province's failure blocking others) but discards the rejection reason for any settled-as-rejected province, making cross-province search failures impossible to diagnose from the console.

Impact is low (telemetry and a secondary search path, not the primary user flow), but failures here are currently invisible during debugging.

**Fix applied in this pass:** added `console.warn` logging on these failure paths, matching the existing `console.warn('[live-ping] ...')` pattern already used elsewhere in `App.jsx`. No behavior change on success paths.

### 🟠 3.3 Potential race condition in geo-lookup-then-ping flow
**File:** `src/App.jsx`, `ensureGeoThenPing`/`fetchGeo` (~lines 958–972)

```js
const ensureGeoThenPing = () => {
  if (geoFetched) { ping(); return; }
  fetchGeo().then((g) => { geo = g; geoFetched = true; ping(); });
};
```

`geo`/`geoFetched` are plain closure variables, not refs tied to a request token. If `ensureGeoThenPing` is invoked twice in quick succession before the first `fetchGeo()` resolves, both calls will independently fetch and the later response simply overwrites the earlier — not corrupting data, but doing duplicate network work and pinging twice. **Not fixed in this pass** — it's intertwined with the broader ping-coalescing logic in `App.jsx` and deserves a focused look rather than a quick patch. Suggested fix: gate with an in-flight promise (`let geoPromise = null; ... geoPromise ||= fetchGeo().then(...)`) instead of a boolean flag.

### 🟡 3.4 CSV import parses large files synchronously on the main thread
**File:** `src/utils/importers.js` (~lines 140–150)

Drillhole CSV parsing chunks rows in batches of 2000 but the chunking itself is still synchronous within a single tick — it doesn't yield to the browser between chunks. For very large files (the app allows up to 50 MB), this can visibly freeze the UI during import. **Not fixed in this pass** (real refactor — recommend moving to a Web Worker, or yielding via `requestIdleCallback`/`setTimeout(0)` between chunks as a smaller interim step).

### 🟡 3.5 Feature-key fallback can collide
**File:** `src/components/MapCanvas.jsx` (~lines 294–297)

```js
const fKey = feature?.id != null ? String(feature.id)
  : feature?.properties?.hole_id || feature?.properties?.holeid
  || feature?.properties?.id || feature?.properties?.name
  || JSON.stringify(feature?.geometry?.coordinates);
```

If none of the id-like properties are present, two distinct features at the exact same coordinates produce the same key and will collide in `featureOverrides` (e.g. one feature's manual style/label override silently applies to the other). **Not fixed in this pass** — `featureOverrides` is persisted, so changing the key scheme needs a migration/compat plan, not a blind edit.

### 🟡 3.6 `eslint-disable` dependency-array suppressions
**File:** `src/App.jsx` (lines ~991-992, ~1002, ~1181, ~1254)

Several `useEffect` hooks suppress `react-hooks/exhaustive-deps` rather than satisfying it. Each one may well be intentional (e.g. only re-running on `user?.id`, not the whole `user` object), but without ESLint actually configured in the repo, these comments are unenforced and easy to drift out of sync with the effect body over time. **Not fixed in this pass** — recommend a case-by-case review once ESLint is added (see §5), rather than guessing intent for each one blind.

## 4. New UI/UX findings

### 🟠 4.1 Drag-and-drop upload zones have no accessible name
**Files:** `src/components/UploadPanel.jsx` (~lines 47–54), `src/components/ClaimsFileUpload.jsx` (~lines 50–58)

Both drop zones are implemented as `<div role="button" tabIndex={0} onKeyDown={...}>` with no `aria-label` and no visible text that doubles as a name — a screen reader announces them as an unlabeled "button". `ClaimsFileUpload.jsx`'s keyboard handler also only responds to `Enter`, not `Space`, which is the more conventional activation key for `role="button"`.

**Fix applied in this pass:** added a descriptive `aria-label` to both drop zones, and added `Space` handling to `ClaimsFileUpload.jsx`'s `onKeyDown`.

### 🟠 4.2 Project thumbnail button has no accessible name
**File:** `src/components/AccountPage.jsx` (`ProjectCard`, lines ~35–41)

```jsx
<button className="acct-card-thumb" type="button" onClick={...}>
  {entry.thumbnail ? <img src={entry.thumbnail} alt="" /> : <span>...</span>}
</button>
```

The `<img>` correctly uses `alt=""` (it's decorative next to the project name below it), but the enclosing `<button>` has no `aria-label`, so when a thumbnail is present the button has **no accessible name at all** — screen reader users get a silent, unlabeled control.

**Fix applied in this pass:** added `aria-label={`Open ${entry.name}`}` to the button.

### 🟡 4.3 Hardcoded colors outside the design system
**Files:** `InsetMap.jsx:22`, `ShadeOverlay.jsx:20`, `AdminPage.jsx:752`, `CalloutsOverlay.jsx:64`

A handful of one-off hex/rgba colors (`#cc2f2f`, `#000000` fallback, an inline `{ png: '#0ea5e9', svg: '#8b5cf6', pdf: '#f59e0b' }` map, a snap-guide `rgba(80,180,255,0.7)`) live outside the app's CSS-variable-based palette. Cosmetic, low priority — flagging for design-system consistency rather than a defect. **Described only.**

### 🟡 4.4 No focus trap in modals; no global `:focus-visible` styling
Modals (`ExportHDModal`, `AuthModal`, `HowToUseModal`, etc.) don't trap focus, so Tab/Shift+Tab can move focus to elements behind the modal while it's open. There's also no project-wide `:focus-visible` outline defined, so the keyboard-focus ring relies entirely on browser defaults, which are inconsistent across the custom-styled buttons/inputs. **Described only** — both are cross-cutting changes worth doing deliberately (focus-trap utility + a shared focus-visible style) rather than patching modal-by-modal.

### 🟡 4.5 Button-style fragmentation
At least 4–5 separate button class families exist with different padding/sizing conventions: `.btn.primary` / `.secondary-btn` / `.btn.compact` / `.landing-nav .btn` / `.topbar-btn`. None are wrong individually, but consolidating toward 1–2 systems (primary/secondary + a compact modifier) would reduce future drift. **Described only** — this is a design decision, not a bug.

### 🟡 4.6 Muted text contrast
Body-sized text in `#94a3b8` on white backgrounds is borderline-to-failing for WCAG AA (≈2.7:1, AA requires 4.5:1 for normal text). Worth a contrast audit pass if accessibility compliance is a goal. **Described only** — palette decision.

### 🟡 4.7 Missing empty states
`LayerList.jsx` shows nothing when there are zero layers (just an empty `<div className="layer-list">`); `AccountPage.jsx` and `ReadOnlyMapStage.jsx` have similar gaps. Not broken, just a missed opportunity to guide new users. **Described only.**

### 🟠 4.9 Bundle size has grown past the prior measurement
Verified with a live `npx vite build` in this pass (output below). Main entry chunk is now 711 kB minified / 184 kB gzip; a `vendor-export` chunk (jsPDF/jszip/html2canvas) adds another 494 kB / 161 kB gzip on top, and a `vendor-geo` chunk (Turf/proj4/d3-geo/topojson) adds 173 kB. Vite also flags two dynamic-import chunking conflicts: `src/export/buildScene.js` and `src/export/renderScene.js` are both dynamically imported from `App.jsx` *and* statically imported elsewhere (`utils/thumbnailCapture.js`, `export/exportPDF.js`, `export/exportPNG.js`, `export/exportSVG.js`), which defeats the intended code-split — those modules end up bundled into the main chunk regardless of the dynamic `import()` in `App.jsx`, because a module that's also statically imported elsewhere can't be moved to a separate chunk by Rollup.

**Not fixed in this pass** (real refactor), but this is a concrete, actionable starting point: removing the static imports of `buildScene.js`/`renderScene.js` from `thumbnailCapture.js` and the `exportPDF/PNG/SVG.js` files (or converting *those* to dynamic imports too) would let Vite actually split the export pipeline out of the main bundle, which is most of the win available here without touching the heavier vendor chunks.

```
dist/assets/vendor-export-CYMnQgm_.js      493.89 kB │ gzip: 160.55 kB
dist/assets/index-D5bfdqk1.js              711.13 kB │ gzip: 184.34 kB
```

### 🟡 4.8 Responsive rough edges
`.map-topbar-title` has a fixed `max-width: 200px` that can truncate project titles unnecessarily on mobile where more width is available; the landing page has no `max-width` constraint below the 480px breakpoint, which can hurt line length/readability on very narrow phones. **Described only** — same family of issue as §2 item 8 (sidebar 46vh on ≤820px), all worth one combined responsive-design pass.

## 5. Infrastructure / process gaps

- **No ESLint.** `eslint-plugin-react-hooks` alone would have caught/flagged the `exhaustive-deps` suppressions in §3.6, and a baseline config would catch unused vars, accidental globals, etc. across a 19K-line JS codebase with zero type-checking.
- **No TypeScript.** Not necessarily a blocker, but worth weighing given the size and the amount of "fallback chain" property access in this codebase (e.g. §3.5).
- **No tests.** 0% coverage. Highest-leverage starting point: the pure utility modules with no DOM/React dependency — `utils/calloutLayout.js` (collision/clamping logic, easy to unit test with synthetic callouts), `utils/claimInfo.js` (string formatting + escaping, exactly the kind of code where a regression like §3.1 should be caught by a test), and `utils/importers.js` (file-format parsing).
- **No pre-commit hooks** to run any of the above even if added.

These are not implemented in this pass (infra investment, separate initiative) but are the single highest-leverage thing the team could add next — every other finding in this report (escaping bugs, dependency-array suppressions, race conditions) is exactly the class of bug that lint + tests catch automatically going forward.

## 6. Suggested priority order for remaining work

1. CSV import off the main thread (Web Worker) — biggest real-world UX hit on large field datasets (§3.4).
2. Feature-key collision fix in `MapCanvas.jsx` (§3.5) — correctness bug, needs a migration plan for persisted `featureOverrides`.
3. Responsive pass: sidebar 46vh + topbar title truncation + landing page narrow-phone max-width (§2 item 8, §4.8) — bundle these into one pass since they're all the same "small screen" theme.
4. Modal focus trap + global `:focus-visible` styling (§4.4).
5. Button system consolidation (§4.5) — design decision first, then mechanical CSS cleanup.
6. Fix the dynamic/static import conflict on `buildScene.js`/`renderScene.js` so the export pipeline actually code-splits out of the main bundle (§4.9) — concrete, already diagnosed, no design decision needed.
7. Stand up ESLint (`eslint-plugin-react-hooks` at minimum) + a small Vitest suite for `utils/`.
