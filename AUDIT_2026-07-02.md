### Reported only (need a design decision or careful testing)

**🟠 1.4 — Wrong-CRS reprojection assumes BC Albers** — `src/hooks/useFileParser.js:27-40`
`needsReprojection()` treats *any* out-of-WGS84-range coordinate as EPSG:3005 (BC Albers). A GeoJSON/KML/loose-shapefile in UTM (e.g. an Ontario NAD83 Zone 17N export, EPSG:26917) gets "reprojected" through the wrong projection and lands silently in the wrong place — no error, features just plot in the ocean or another province. (Zipped shapefiles with a `.prj` are safe — `shpjs` reads the projection itself.)
*Suggested fix:* after reprojection, sanity-check the result lands within plausible bounds (e.g. Canada + buffer); if not, show a "couldn't detect this file's coordinate system" error with a CRS picker (proj4 already ships; a dropdown of the ~10 common Canadian UTM zones + provincial grids covers nearly all real uploads).

**🟡 1.5 — Coordinate-precision trim rewrites user text** — `src/utils/projectStorage.js:11-13`
`trimCoordinatePrecision()` runs its `(-?\d+\.\d{7,})` regex over the **entire serialized JSON**, including string values. A callout or label containing a long decimal — `"Au 1.2345678 g/t over 3.5 m"` — is silently rewritten to `1.234568` on every local save. Real coordinates are the target; user text is collateral.
*Suggested fix:* trim during `JSON.stringify` with a replacer that only touches `geometry.coordinates` arrays, or run the regex only over layer geojson before assembly.

**🟡 1.6 — `ensureGeoThenPing` race** (carried over from first audit, still open) — `App.jsx:958-972`. Coalesce with a shared promise (`geoPromise ||= fetchGeo()`) instead of a boolean.

**🟡 1.7 — Distance-line label background is fixed-width** — `AnnotationOverlay.jsx:361-365`. The white rect behind the measurement is hardcoded 56px; labels like `1609.3 km` overflow it. Measure text or size the rect from `label.length`.

**🟡 1.8 — `formatRelativeDate` duplicated** — near-identical copies in `LandingPage.jsx:16` and `AccountPage.jsx` (`fmtRelative`). Extract one into `src/utils/`.

### Reviewed and clean (for the record)
`api/claims.js` (excellent defensive proxy: field-name allowlisting, SQL escaping, HTML-block detection, esri→GeoJSON fallback, retry with backoff), `api/bc-claims.js`, `api/geo.js`, `useAuth.jsx`, `SharedMapViewer.jsx`, `cloudStorage.js`, `projectStorage.js` (aside from 1.5), export pipeline entry points.

## 2. Spacing & design issues (measured, not vibes)

Quantified from `styles.css` (2,676 lines):

| Metric | Current state | Target |
|---|---|---|
| Distinct hex colors | **164**, zero CSS custom properties | ~12–16 tokens (`--ink`, `--muted`, `--accent`, `--surface`, `--border`, semantic green/red…) |
| Font sizes | **20 distinct px values**, incl. 7px, 9px, 17px oddballs | 8-step scale (10/11/12/13/14/16/20/28) |
| Border radii | 10 values (3,4,5,6,7,8,10,12,14,999) | 4 (4/8/12/999) |
| Off-grid spacing | 96 declarations using 5/7/9/13/15/17/18px | 4px grid (4/8/12/16/24) |
| `:focus-visible` rules | **0** — only text inputs have focus styles | One global rule (see below) |

Specific issues:

- **2.1 🟠 No visible keyboard focus on buttons/cards.** No `outline: none` reset exists so browser defaults survive, but they clash with the polished input focus rings (`box-shadow: 0 0 0 3px rgba(77,134,213,0.14)`). One rule fixes the whole app: `:focus-visible { outline: 2px solid #2563eb; outline-offset: 2px; }`.
- **2.2 🟠 Muted text `#94a3b8` used 63×** — 2.9:1 on white, fails WCAG AA (4.5:1) at body sizes. Darkening one value to `#64748b` (4.75:1) fixes all 63 uses when tokenized — the strongest argument for doing tokens first.
- **2.3 🟡 Modal padding drift:** 28px (ExportHD) / 24px (HowToUse) / 20px (nav) — pick one.
- **2.4 🟡 Input focus ring colors drift:** `#4d86d5` vs `#3b82f6` vs `#2563eb` across 8 `:focus` rules — same token problem.
- **2.5 🟡 `.map-topbar-title` fixed `max-width: 200px`** truncates titles even when the topbar has room; use `flex: 1; min-width: 0` on the parent instead.
- **2.6 🟡 Sidebar takes 46vh at ≤820px** (carried over, still the weakest responsive behavior). Recommend a bottom-sheet pattern: collapsed 56px bar with drag handle, map gets the screen.

**Suggested design improvements (highest leverage first):**
1. Introduce CSS custom properties for color/radius/spacing — mechanical find-and-replace, huge future payoff, zero visual change by construction.
2. Add the global `:focus-visible` rule (one line, accessibility win).
3. Fix muted-text contrast via the token.
4. Consolidate the 5 button families (`.btn`, `.secondary-btn`, `.topbar-btn`, `.landing-nav .btn`, `.landing-hero2-primary`) into `.btn` + variants (`--primary/--secondary/--ghost`, `--sm/--md/--lg`).
5. Empty states for `LayerList` ("No layers yet — search the registry or upload a file") and `AccountPage`.

## 3. Landing page & copy review

**What's working well (keep):** the before/after panel is the single best selling element on the page; "Three steps. Under five minutes." is concrete; the FAQ pre-empts real objections honestly; the collapsible data-sources directory is genuinely useful and good for SEO; per-card "Try this style →" with sample data is a strong low-friction hook; click-tracking with `data-track` is already wired for measuring all of this.

**Issues, in priority order:**

- **3.1 🟠 The hero says the same thing twice.** Tagline "Raw data in. Investor map out." is immediately followed by h1 "Turn raw exploration data into clean investor maps." — identical message, adjacent lines. Keep the h1; either drop the tagline or repurpose that line as proof (e.g. "**2,400+ maps generated** · Search 7 provincial registries live" — you already have `page_views`/`export_events` data to back a real number).
- **3.2 🟠 Competing CTAs with mismatched verbs.** Nav says "Try for Free", hero primary says "Search the Claims Registry", secondary row says "Upload a Shapefile" / "See an Example", bottom says "Open the Editor". Four different entry framings. Also "Try for **Free**" implies a paid tier exists — the FAQ says everything's free. Recommend: nav CTA "Make a map", keep hero as-is (registry search is the right primary — it's the differentiator), bottom CTA mirrors hero.
- **3.3 🟡 Province list crammed into the primary button.** The sub-line "By name or claim number · BC · Ontario · Quebec · Saskatchewan · Manitoba · Newfoundland · Yukon" is 100+ characters inside a button. "By company name or claim number — 7 provinces, live" reads faster; the full list already appears in "What's included".
- **3.4 🟡 Free/email tension.** "Free · No account" trust strip, then FAQ: "Watermark-free exports just ask for your email." Honest, but the strip overpromises slightly. Suggest FAQ wording: "Yes — everything is free. Exports are watermark-free with just an email (no account needed)."
- **3.5 🟡 Asymmetric before/after interaction.** The "after" panel is a clickable live demo; the "before" panel is inert. Make the whole before/after block one click target for the demo.
- **3.6 🟡 No social proof anywhere.** Even one line — a testimonial, a logo strip, or a real usage counter — would materially help a conversion page this good otherwise.
- **3.7 🟡 Naming nit:** the `onSearchBCClaims` prop opens the all-province registry search — rename to `onSearchClaims` when convenient.

## 4. Feature opportunities (prioritized)

Ranked by user value ÷ effort, given what the codebase already has:

| # | Feature | Why it fits now | Effort |
|---|---------|----------------|--------|
| F1 | **Undo/redo** (Ctrl+Z) | Editor has none. Project state is already a single serializable object (`project`) — a bounded history stack of ~30 snapshots (diff- or structural-sharing-based) in App.jsx covers markers, callouts, styles, layout. Biggest everyday UX gap. | M |
| F2 | **Claim-expiry intelligence** | `GOOD_TO_DATE` is already parsed and clustered. Add "⚠ 3 claims expire within 90 days" badges on layers/groups, an expiry column sort, and optionally a signed-in email reminder (Supabase cron). Directly monetizable; no competitor in this niche does it well. | S–M |
| F3 | **Assay/intercept auto-callouts** | CSV importer + ColumnMapper already exist; drillhole markers exist; callouts exist. Map grade columns (`Au g/t`, `interval`) at import → auto-generate ranked "12.4 g/t Au over 6 m" callouts for top-N intercepts. This is *the* drill-results-map workflow. | M |
| F4 | **Saved-search change alerts** ("claim watch") | The QC weekly-refresh pipeline (`update-qc-claims.yml`) proves scheduled jobs work. Let a signed-in user save a company/area search; diff weekly; email "New claims staked near Project X". Strong retention + account-creation driver. | M–L |
| F5 | **Fixed-scale export** (e.g. exactly 1:50,000) | Export pipeline + scale bar already compute scale; add a target-scale input that back-solves zoom/dimensions. Geologists need this for technical reports; supports the NI 43-101 template story. | S–M |
| F6 | **Cross-section/strip-log view** | CSV synonyms already parse `azimuth`/`dip` — the data's there. Section line drawn on map → simple projected section view of collars/traces. Large effort, large differentiation. | L |
| F7 | **DXF/GeoTIFF export** | SVG scene graph exists (`renderScene.js`); DXF is a mechanical transform of it. CAD handoff is a common ask from engineering-adjacent users. | M |
| F8 | **Org/team brand kits** | `templates` table + RLS exist; add an `org_id` and sharing. Natural first paid tier alongside F2/F4. | M |

Suggested sequence: **F1 → F2 → F3** (each lands independently), with F4 as the retention play once accounts matter.
